# DailyNotes 写作共识

> 本文档约定 DailyNotes 仓库的笔记写作规范，确保内容风格一致、知识可复用。

---

## 核心原则：只收录可复用的知识资产

仓库定位为**个人知识库**，每篇文章应该是一份"可反复查阅的知识资产"，而不是"一次性的经历记录"。

**适合收录的四种品类：**

| 品类 | 说明 | 示例 |
|------|------|------|
| 外部知识消化笔记 | 对教程、文档、论文的翻译/整理/删补/注释 | `Java内存马`、`Powershell-Empire`、`Pinia` |
| 操作指南/教程 | 面向复用的方法总结，可通用化而非绑定一次操作 | `Web漏洞通用型Python复现脚本编写指南` |
| 速查手册 | 按条目排列的语法/配置/技巧合集 | `Note-python.md`、`Markdown.md` |
| **项目文档** | 特定项目的介绍、功能、技术栈、部署方法等 | `infinite-canvas/infinite-canvas.md` |

**不适合：** 纯时间线叙事的一次性排障记录（"某年某月在某服务器上 build 某项目时遇到了某错误"）。这类内容除非能提炼出可复用的方法论，否则不建议单独成文。

---

## 结构约定

### 1. 外部引用优先

如果是基于外部资料整理的笔记，**在文章开头用 blockquote 标注来源**：

```markdown
> [一文看懂内存马 - FreeBuf网络安全行业门户](https://www.freebuf.com/articles/web/274466.html)
```

如果是多来源，全部列在开头：

```markdown
> [Introduction | Pinia (vuejs.org)](https://pinia.vuejs.org/introduction.html)
>
> [技术胖-Pinia 入门视频教程](https://jspang.com/article/82)
```

### 2. 按概念/主题组织，而非按时间线

- 使用层级标题（`##` → `###` → `####`）组织知识点
- 每个 section 信息自足，允许读者跳跃查阅
- 避免"背景 → 现象 → 原因 → 修复"的时间线叙事

### 3. 目录导航

超过 30 行的文章在标题下方生成目录（VuePress 的 `[[toc]]` 或手动 TOC）。

### 4. Frontmatter

每篇文章必须有 frontmatter：

```yaml
---
category:
  - 主分类
  - 子分类
tags:
  - 标签1
  - 标签2
excerpt: 一句话概述文章内容，用于索引和搜索。
---
```

## 语气约定

- **声明式**，而非叙事式
  - ✅ "使用 `useSearchParams()` 时需在 Suspense 边界内"
  - ❌ "这次 build 时遇到了 useSearchParams 的错误"

- **第一人称仅限于表达选择偏好**（在指南类文章中），不用于讲述个人经历
  - ✅ "个人平时使用的是 Python 3.8 及以上版本"
  - ❌ "这次我在 TencentSouel 上 clone 了项目"

- **避免时间锚点**：不出现"这次"、"当时"、"今天"等词

---

## 格式约定

- **代码块**标注语言：```` ```go ```、```` ```bash```、```` ```text ````
- **内联代码**用反引号包裹
- **图片**统一使用 CDN 路径：`http://cdn.ayusummer233.top/DailyNotes/...`

  例外：语法演示场景（如教 Markdown 图片语法时），相对路径是演示素材本身，可用 `![](./相对路径.png)`。
- **技术栈表格**：当一个页面需要展示项目的技术栈时，优先使用表格而非列表，方便快速对照：

  ```markdown
  | 层次 | 技术 |
  |------|------|
  | 框架 | Next.js 16.2 (App Router)、React 19.2 |
  | UI | Ant Design 6、Tailwind CSS 4 |
  | 状态管理 | Zustand 5 |
  ```

- **分隔线** `---` 仅用于分隔大段，小节之间不需要
